1. 架构总览

系统由四层组成，Buglog 与知识库共享通用工具层：

Hook 脚本层

  • kb-inject.js（SessionStart）
  • bug-detect-user-report.js
  • bug-capture.js
  • bug-failure-capture.js

共享工具层

  • hook-utils.js（通用：readStdin/outputContext）
  • bug-utils.js（bug 专用：scanBugs/updateStats）
  • kb-utils.js（知识库：generateKnowledgeMap）

数据持久层

  • .buglog/bugs/*.md + index.md + stats.json
  • .failure_counts.json /.edit_counts.json（临时）
  • ~/.claude/rules/*.md（规则层）
  • Obsidian vault（知识存储）

1.2 Layer 分层

Layer机制内容
Rules 层~/.claude/rules/*.md 会话启动时原生加载确定性规则（分类：通用/前端/后端/游戏）
知识库地图层SessionStart hook 扫描 Obsidian vault 注入精简索引分类 -> 文件名 -> 一句话描述
Buglog 层PostToolUse/PostToolUseFailure/UserPromptSubmit hook 检测项目 bug 记录、通用命令错误

2. 文件组成与职责

文件Hook 事件Matcher职责
hook-utils.js — — 通用 readStdin() / outputContext()，所有 hook 共享
bug-utils.js — — Bug 专用：scanBugs / updateStats / updateIndex / getNextBugId / recordFailure / getEscalationMessage（count<2 返回 null）
kb-utils.js — — 知识库专用：parseKbIndex() 解析 Obsidian claude.md 索引 / generateKnowledgeMap() 生成精简地图
kb-inject.js SessionStart * 会话启动时生成知识库地图，通过 additionalContext 注入到会话上下文
bug-detect-user-report.js UserPromptSubmit * 检测辱骂关键词 -> 自动写入；检测 bug 关键词 -> 修复指引
bug-capture.js PostToolUse Edit|Write 追踪编辑次数，>=2 次或 >=5 文件时触发提醒
bug-failure-capture.js PostToolUseFailure * Bash 命令追踪重复失败。count=1: 静默 / count 2-3: 警告 / count>=4: 强烈警告

5. 知识库读写机制

5.1 会话启动：知识地图注入

每次会话启动时，kb-inject.js（SessionStart hook）扫描 Obsidian 知识库的 claude.md 索引，生成精简地图通过 additionalContext 注入。

注入内容示例：

[知识库地图]
开发环境 - 工具链管理(nvm/Git配置)、命令行环境(Git Bash/WSL/Docker/tmux)
Claude Code - Hooks钩子系统、Buglog系统、skill管理系统
AI与LLM - Agent框架(langchain/langgraph)、RAG系统
...

遇到相关问题时，读取 D:/ASUS/Documents/obsidian/new/ 下对应目录的文件。
读取前先判断时效性（模型能力 + 必要时联网验证）。
使用知识后，更新文件的 ref_count frontmatter（+1）。
过时知识不要使用，采用新方案后更新知识库文件。

5.2 时效性判断流程

遇到问题，查地图找到相关文件
|
读取知识库文件内容
|
判断时效性
相关且有效
使用知识 + ref_count++
相关但过时
新方案解决 + 更新知识库 + ref_count++
无关
自己解决 -> 记录新知识

5.3 Rules 层

知识库 rules/ 目录按领域分类（通用/前端/后端/游戏），文件复制到 ~/.claude/rules/ 下，由 Claude Code 在会话启动时原生加载。

Rules 增长来源：

  • 通用 Bug 抽取（主要）：bug 记录到 buglog -> /buglog close 时 AI 评估 -> 人工确认 -> 提炼写入 rules/
  • 知识升级（未来）：ref_count 高的知识条目半自动升级为 rule
  • 手动添加：直接编辑 rules 文件

5.4 /recall 手动检索

/recall <关键词> 技能在 Obsidian 知识库中搜索相关内容：

  1. 读 claude.md 索引，关键词匹配目录/文件名
  2. 定位最相关的 1-3 个文件并读取
  3. 判断时效性后展示结果 + ref_count++

5.5 参考次数

知识库文件的 frontmatter 中包含 ref_count 字段，每次被参考 +1：

---
ref_count: 5
---

用于识别最有价值的知识，判断哪些内容需要更新或清理。

5.6 数据流总览

会话启动
|
Layer 1: ~/.claude/rules/*.md 自动加载
Layer 2: SessionStart hook 注入知识地图
|
工作中遇到问题
|
查地图 -> 定位文件 -> 读取 -> 判断时效性
命令失败 -> hook 检测 -> count>=2 提醒 + 建议查知识库
/buglog close -> 评估通用性 -> 抽取 rule
|
会话结束
|
有价值知识自动沉淀到知识库文件
|
下次会话：新沉淀的知识被地图索引覆盖

6. 输入与输出

5.1 各 Hook 的输入（stdin JSON）

Hook 事件stdin 字段关键字段说明
UserPromptSubmit { "prompt": "..." } 用户提交的完整消息文本
PostToolUse { "tool_name": "...", "tool_input": {...} } tool_name: "Edit"/"Write"；tool_input.file_path
PostToolUseFailure { "tool_name": "...", "tool_input": {...}, "error": "..." } error: 错误描述文本

5.2 Hook 输出协议

输出格式用途
{ "hookSpecificOutput": { "hookEventName": "...", "additionalContext": "..." } } 通过 additionalContext 向 Claude 注入提醒消息

5.3 副作用输出

操作执行者写入位置
创建 bug 文件bug-detect-user-report.js（辱骂时自动创建）、/buglog add（手动）.buglog/bugs/*.md
标记 bug 为 fixed/buglog close（AI 手动调用）.buglog/bugs/*.md（改 frontmatter）
更新统计写入 bug 后.buglog/stats.json
更新索引写入 bug 后.buglog/index.md
失败计数（临时）bug-failure-capture.js.buglog/.failure_counts.json
编辑计数（临时）bug-capture.js.buglog/.edit_counts.json

7. Hook 注册配置

所有 hook 在 ~/.claude/settings.json 的 hooks 字段中注册：

"hooks": {
  "SessionStart": [{
    "matcher": "",
    "hooks": [{ "type": "command", "command": "node \"$HOME/.claude/hooks/kb-inject.js\"", "timeout": 5 }]
  }],
  "UserPromptSubmit": [{
    "matcher": "",
    "hooks": [{ "type": "command", "command": "node \"$HOME/.claude/hooks/bug-detect-user-report.js\"", "timeout": 5 }]
  }],
  "PostToolUse": [{
    "matcher": "Edit|Write",
    "hooks": [{ "type": "command", "command": "node \"$HOME/.claude/hooks/bug-capture.js\"", "timeout": 3 }]
  }],
  "PostToolUseFailure": [{
    "matcher": "",
    "hooks": [{ "type": "command", "command": "node \"$HOME/.claude/hooks/bug-failure-capture.js\"", "timeout": 3 }]
  }]
}

9. 运行示例

9.1 场景：同一命令连续失败（新逻辑）

# 第 1 次失败
（静默，不发送任何通知）

# 第 2 次失败
[Buglog] [!] 同一命令连续失败 2 次，请评估是否需要记录 bug。
命令: python scripts/mcp_fetcher.py --check
错误: Exit code 49

# 第 4 次失败
[Buglog] [!!] 同一命令连续失败 4 次！强烈建议调用 /buglog add 记录此问题。
命令: python scripts/mcp_fetcher.py --check
错误: Exit code 49
如果这不是 bug，请忽略。

9.2 场景：用户骂人

用户: "你太蠢了，这么简单都搞不定"

[Buglog] 检测到用户表达不满，已自动记录为 BUG-006。
请认真反思本次会话中可能存在的问题，并在 bug 记录的"改进措施"部分填写具体改进方案。
用户的原始消息: "你太蠢了，这么简单都搞不定"

→ 自动创建 .buglog/bugs/2026-07-21-user-frustration-.md
→ source: user-frustration, severity: minor

9.3 场景：用户指出遗漏问题

用户: "还是有问题，你没有发现这次对话中有很多错误"

[Buglog] 用户报告了一个 bug。请：
1. 先确认你理解了用户描述的问题
2. 定位根因后再修复
...
另外，用户可能在指出本次会话中遗漏的问题。请回顾本次会话中是否有多次失败、
报错、用户纠正的情况，如果有被遗漏的问题，调用 /buglog add 逐条记录。

9.4 场景：标记 Bug 为已修复并抽取 Rule

AI: /buglog close BUG-001

-> 更新 .buglog/bugs/2026-07-21-python-exit-code-49.md
  status: open -> fixed
  添加 fixed_at: 2026-07-22T10:00:00
-> 重新生成 index.md + stats.json
-> AI 评估：此 bug 涉及 Windows Python 命令兼容性，有通用价值
-> 提醒用户："这个 bug 是通用的（Windows py vs python3），要抽取为 rule 吗？"
-> 用户确认后写入 ~/.claude/rules/通用.md + Obsidian rules/通用.md
